RDevents — полное техническое задание на доработку платформы
Единый документ по пользовательскому кабинету, профилю, командам, календарю, экспорту и восстановлению пароля
Проект	RDevents
Формат документа	Единое техническое задание на пакет доработок
Основные темы	Password reset, visibility полей профиля, новый cabinet dashboard, freeze команд, экспорт, аватарки, календарь
Назначение	Основа для реализации, code review, QA и приёмки

Документ собран как единый consolidated spec: он объединяет существующую архитектуру RDevents, новую переработку кабинета, требования к командам, календарям, экспортам и полный блок по скрытию/отображению полей профиля.
 
1. Назначение документа
Настоящее техническое задание объединяет в одном документе все согласованные доработки проекта RDevents. Документ должен использоваться как единая основа для проектирования, реализации, тестирования и приёмки изменений без разрыва между продуктовой постановкой, UX-логикой, API, базой данных и административными сценариями.
Документ опирается на текущую архитектуру проекта: monorepo с Next.js frontend в apps/web, Express + Prisma backend в services/api, PostgreSQL как основной БД, существующие модули auth, events, admin, registrations, uploads, analytics, support, а также на отдельное техническое задание по управлению видимостью полей профиля.
Цель пакета доработок: превратить текущий личный кабинет и event workflow в быстрый рабочий интерфейс участника и капитана команды, усилить операционные инструменты админки, закрыть базовые product gaps и подготовить платформу к реальной эксплуатации на мероприятиях.
2. Ключевые изменения пакета
•	Добавить полноценное восстановление и сброс пароля по email.
•	Включить в общее ТЗ платформенную подсистему скрытия и показа полей профиля из отдельного файла требований.
•	Полностью переработать маршрут /[locale]/cabinet: по умолчанию показывать не профиль, а рабочий дашборд участника по текущему мероприятию.
•	Если у пользователя несколько мероприятий, дать быстрый выбор активного события и мгновенное переключение рабочего пространства.
•	Усилить управление командой для капитана: удаление участников, редактирование названия и описания, но только до отправки заявки/сабмита.
•	После отправки командной заявки зафиксировать состав и ключевые поля команды; дальнейшие изменения — только по контролируемому сценарию.
•	Переместить фотографию из общего профиля в раздел регистрационных данных.
•	Добавить удобный экспорт данных мероприятия из админки с настраиваемыми конфигурациями выгрузки.
•	Добавить выгрузку всех аватарок и отдельный пакет материалов для изготовления «паспорта игры».
•	Сделать нормальный календарь как на публичном сайте, так и в админке.
•	Добавить дополнительные полезные функции для участника: быстрые статусы, дедлайны, приглашения, документы, чек-листы, ссылки на команду, историю решений администратора.
3. Продуктовые принципы, обязательные для реализации
•	Backend является источником истины. Нельзя реализовывать критическую бизнес-логику только на фронтенде.
•	Все новые пользовательские сценарии должны иметь быстрый путь: минимальное число кликов, минимум переходов между страницами, мгновенный обзор статуса мероприятия.
•	Новые возможности не должны ломать существующую архитектуру monorepo, Prisma schema, auth/session flow и существующие event модели.
•	Все ограничения по ролям и состояниям сущностей должны контролироваться сервером.
•	Для всех операций, влияющих на состав участников, команды, профиль и восстановление доступа, нужно предусмотреть понятные ошибки, аудит и тесты.
•	Публичная часть, личный кабинет и админка должны оставаться двуязычными: ru/en.
4. Текущий контекст проекта, который нужно сохранить
В текущем проекте уже есть: email-регистрация в три шага, access/refresh token flow, отдельные маршруты /api/auth, /api/events, /api/me, /api/admin, event memberships, volunteers, teams, invitations, analytics, support и media uploads. Существуют пользовательские маршруты для получения списка своих событий, команд, приглашений и отдельный workspace по событию.
Сейчас маршрут /[locale]/cabinet по сути является alias профиля. Это поведение нужно изменить: профиль не должен исчезнуть, но должен стать отдельным разделом, а не дефолтным экраном пользователя.
Также уже существует командный контур: создание команды, обновление команды, join, leave, remove member, transfer captain, submit team for approval, approval/rejection pending members. Это надо не переписывать с нуля, а аккуратно расширить и перестроить UX поверх существующей архитектуры.
5. Итоговая карта пользовательских сценариев после доработки
После внедрения изменений у системы должны быть следующие главные сценарии.
•	Незалогиненный пользователь может запросить восстановление пароля, подтвердить токен из письма и задать новый пароль.
•	Обычный пользователь после логина попадает в /cabinet и видит рабочую панель по текущему мероприятию, а не форму профиля.
•	Если мероприятий несколько, пользователь выбирает активное событие через быстрый селектор и сразу попадает в соответствующее рабочее пространство.
•	Участник видит статус участия, дедлайны, требования профиля, event-specific требования, состояние команды, приглашения и доступные действия.
•	Капитан команды управляет своей командой до сабмита; после сабмита состав фиксируется.
•	Администратор настраивает видимость полей профиля, обязательные поля события, календарные представления и конфигурируемые выгрузки.
•	Администратор может выгрузить агрегированный пакет мероприятия, включая участников, команды, волонтёров, документы статусов, аватары и отдельный комплект для «паспорта игры».
6. Подсистема восстановления и сброса пароля
В системе необходимо добавить полноценный recovery flow для пользователей, зарегистрированных по email/password. Подсистема должна работать независимо от текущего flow регистрации, но использовать общую инфраструктуру email delivery, rate limiting, audit logging и безопасное хранение хешей токенов.
6.1. Цели
•	Пользователь может запросить письмо для восстановления доступа.
•	Пользователь может перейти по одноразовой ссылке или использовать токен/код из письма.
•	Пользователь может задать новый пароль без знания старого.
•	После смены пароля все старые refresh-сессии должны быть отозваны, кроме опционального текущего сценария, если это отдельно согласовано. По умолчанию — отозвать все.
•	Flow должен быть безопасным против enumeration, brute force и reuse.
6.2. UX-flow
1.	На странице входа добавить ссылку «Забыли пароль?» / «Forgot password?»
2.	Экран запроса email: пользователь вводит email.
3.	Система всегда отвечает нейтрально: «Если аккаунт существует, письмо отправлено».
4.	Пользователь получает email со ссылкой вида /[locale]/reset-password?token=...
5.	Страница сброса проверяет токен, показывает форму нового пароля и подтверждения.
6.	После успешной смены пароля пользователь получает подтверждение, а затем либо автоматически логинится, либо отправляется на страницу входа. Рекомендуемый вариант для production: редирект на login с сообщением об успехе.
6.3. Backend-модель
Добавить отдельную Prisma-модель, например PasswordResetRequest или PasswordResetToken.
model PasswordResetToken {
  id            String   @id @default(cuid())
  userId        String
  tokenHash     String   @unique
  expiresAt     DateTime
  usedAt        DateTime?
  requestedAt   DateTime @default(now())
  requestedIp   String?
  requestedUa   String?
  consumedIp    String?
  consumedUa    String?

  user          User     @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@index([userId, expiresAt])
  @@index([expiresAt])
  @@map("password_reset_tokens")
}

6.4. API
•	POST /api/auth/password/forgot — принимает email, создаёт reset request и отправляет письмо.
•	POST /api/auth/password/verify — опциональный endpoint для проверки валидности токена.
•	POST /api/auth/password/reset — принимает token + newPassword, завершает сброс.
•	Опционально для admin audit: GET /api/admin/auth/password-reset-activity.
6.5. Безопасность
•	Хранить только hash токена, не хранить токен в открытом виде.
•	Время жизни токена: по умолчанию 30 минут.
•	Токен одноразовый.
•	После успешной смены пароля отозвать все refresh sessions пользователя.
•	Лимит на forgot password запросы по IP и по email.
•	Нельзя раскрывать, существует ли email в системе.
•	Нельзя использовать reset flow для social-only аккаунтов без passwordHash, пока им не задан локальный пароль; для них должна быть понятная ошибка или отдельная логика привязки пароля.
6.6. Email
•	Добавить email template: password_reset_requested.
•	Добавить template variables: userName, resetUrl, expiresAt, ipHint, supportContact.
•	Добавить отдельный audit event: PASSWORD_RESET_REQUESTED, PASSWORD_RESET_COMPLETED, PASSWORD_RESET_REJECTED.
6.7. Acceptance criteria
•	Пользователь может восстановить доступ без обращения к администратору.
•	Токен нельзя использовать повторно.
•	После сброса старые сессии недействительны.
•	Flow одинаково работает на desktop/mobile и на ru/en.
7. Подсистема управления видимостью полей профиля
Данный раздел интегрирует отдельное техническое задание по управлению отображением полей профиля и является обязательной частью общего пакета. Реализация должна быть платформенной, централизованной и серверно-управляемой.
7.1. Проблема
Сейчас часть полей профиля жёстко зашита и дублируется в backend и frontend. Из-за этого скрыть поле только во фронтенде недостаточно: пользователь сможет отправить payload вручную, админская форма события сможет ссылаться на скрытое поле, а flow регистрации станет рассогласованным.
7.2. Цель
SUPER_ADMIN / PLATFORM_ADMIN должен иметь возможность централизованно управлять тем, какие поля показываются пользователю в /cabinet/profile и в связанных user-facing сценариях. Скрытые поля не удаляются из БД, но не отображаются и не редактируются пользователем.
7.3. Обязательные эффекты
•	Скрытое поле не отображается в пользовательском интерфейсе профиля.
•	Скрытое поле не принимается пользовательскими API профиля.
•	Скрытое поле не может быть выбрано как requiredProfileField события.
•	Если поле уже используется в requiredProfileFields, скрытие должно быть заблокировано сервером.
•	Администратор продолжает видеть конфигурацию поля и может вернуть его обратно.
7.4. Единый реестр полей
Нужно ввести единый registry profile fields как source of truth. Реестр должен хранить key, sectionKey, type, labelRu, labelEn, defaultVisibleInCabinet, allowEventRequirement, isCompositeRequirement, storageScope. Рекомендуемое размещение: packages/shared/src/profile/profile-field-registry.ts.
7.5. Таблица конфигурации
model ProfileFieldVisibilitySetting {
  key                String   @id
  sectionKey         String
  isVisibleInCabinet Boolean  @default(true)
  updatedByUserId    String?
  createdAt          DateTime @default(now())
  updatedAt          DateTime @updatedAt

  updatedByUser      User?    @relation(fields: [updatedByUserId], references: [id], onDelete: SetNull)

  @@index([sectionKey])
  @@map("profile_field_visibility_settings")
}

7.6. Минимальный перечень полей в реестре
В реестр нужно завести как минимум поля из registration_data, general_info, personal_documents, contact_data, activity_info, включая lastNameCyrillic, firstNameCyrillic, middleNameCyrillic, lastNameLatin, firstNameLatin, middleNameLatin, birthDate, gender, citizenshipCountryCode, residenceCountryCode, phone, telegram, consentPersonalData, consentMailing, avatarUrl/avatarAssetId, nativeLanguage, communicationLanguage, региональные поля адреса, document completion flags, social links, activity status, organizationName, facultyOrDepartment, classCourseYear, positionTitle, additionalLanguages, achievementsText, emergencyContact и др.
7.7. Новое расположение фотографии
В рамках этого же пакета поле с фото пользователя нужно логически перенести из общего блока general_info в registration_data. На уровне реестра и фронтенда avatar должен считаться частью регистрационных данных, так как он критически участвует в идентификации участника и в последующих выгрузках для игровых паспортов.
7.8. Admin API
•	GET /api/admin/profile-fields — вернуть полный список полей с метаданными, видимостью и информацией об использовании в событиях.
•	PATCH /api/admin/profile-fields/:key — изменить видимость одного поля.
•	PATCH /api/admin/profile-fields — bulk update нескольких полей.
•	Доступ только для PLATFORM_ADMIN и SUPER_ADMIN.
7.9. Серверная фильтрация
•	Обновить GET /api/auth/me, GET /api/auth/profile/sections, PATCH /api/auth/profile/sections/:sectionKey и связанные helpers.
•	Скрытые поля нельзя silently принимать в PATCH; рекомендовано возвращать 400 PROFILE_FIELD_HIDDEN.
•	Summary незаполненных полей и section completion logic должны учитывать только видимые поля.
7.10. Event compatibility
•	В requiredProfileFields события могут попадать только поля из registry, у которых allowEventRequirement=true и isVisibleInCabinet=true.
•	Если поле скрывается, а событие ещё требует его, backend должен вернуть 409 PROFILE_FIELD_IN_USE с деталями по событиям.
7.11. Frontend
•	В /cabinet/profile секции должны строиться на основе fieldVisibility config, а не на основе жёстко вшитого списка.
•	В админке добавить страницу /[locale]/admin/profile-fields с поиском, фильтрами, статусами visible/hidden/in-use и подтверждением действия.
•	В форме создания/редактирования события selector requiredProfileFields должен брать данные из backend, а не из hardcoded констант.
7.12. Миграция
•	Создать таблицу profile_field_visibility_settings.
•	Проставить для всех текущих полей visible=true.
•	Проверить консистентность Event.requiredProfileFields относительно registry.
7.13. Acceptance criteria
•	Админ может скрыть/вернуть поле.
•	Скрытое поле исчезает из кабинета пользователя.
•	Пользователь не может изменить скрытое поле через API.
•	Скрытое поле нельзя выбрать обязательным полем события.
•	Если поле используется событием, backend запрещает скрытие.
8. Новый пользовательский кабинет и event dashboard
Дефолтный маршрут /[locale]/cabinet больше не должен рендерить страницу профиля. Вместо этого он должен стать точкой входа в рабочее пространство пользователя по текущему мероприятию.
8.1. Новая информационная архитектура кабинета
•	/[locale]/cabinet — event dashboard по текущему или выбранному событию.
•	/[locale]/cabinet/profile — профиль и регистрационные данные.
•	/[locale]/cabinet/events — список всех мероприятий пользователя.
•	/[locale]/cabinet/invitations — входящие приглашения в команды.
•	/[locale]/cabinet/support — обращения и коммуникация.
8.2. Правила выбора активного события
7.	Если у пользователя нет ни одного события и ни одной заявки, показывать onboarding-empty state с CTA на каталог мероприятий.
8.	Если есть одно текущее релевантное событие, сразу открывать его workspace без промежуточного выбора.
9.	Если событий несколько, показывать быстрый selector активного события сверху страницы; последний выбор хранить в user preference/local cache.
10.	Текущим считается событие, которое: а) ещё не завершено; б) имеет наиболее высокий приоритет по статусу участия; в) при равенстве — ближайшее по startsAt.
8.3. Что обязательно показывать на dashboard
•	Карточка события: название, даты, место, статус события.
•	Мой статус: PARTICIPANT / VOLUNTEER / EVENT_ADMIN / PENDING / APPROVED / REJECTED / RESERVE и т.д.
•	Статус регистрации и прохождения требований.
•	Статус команды: есть ли команда, роль в команде, размер, сабмит, блокировки, pending members, pending invitations.
•	Блок дедлайнов: регистрация, сабмит команды, загрузка документов, check-in, начало мероприятия.
•	Блок обязательных действий: заполнить профиль, подтвердить участие, принять приглашение, завершить регистрационные данные, загрузить фото/документы.
•	Быстрые действия: открыть карточку команды, принять/отклонить приглашение, перейти к недостающим полям профиля, открыть календарь события, скачать материалы.
•	История решений администратора по заявке и команде.
8.4. Полезные дополнительные функции, которые следует включить
•	Invitation inbox прямо на dashboard без перехода на отдельную страницу.
•	Progress bar готовности к участию: профиль, фото, документы, команда, event form, подтверждения.
•	Event checklist с галочками и подсветкой блокеров.
•	Краткая лента уведомлений по мероприятию: команда отправлена, заявка одобрена, дедлайн изменён, админ запросил корректировку.
•	Ссылки на материалы участника: регламент, памятка, карта, чат, support.
•	Кнопка «Сменить активное событие» всегда доступна без ухода со страницы.
8.5. API для кабинета
Нужно ввести агрегированный endpoint, чтобы не собирать dashboard из множества разрозненных вызовов.
•	GET /api/me/dashboard — вернуть summary по всем событиям пользователя и activeEventId.
•	GET /api/me/events/:slug/workspace — расширить текущий endpoint до полного event workspace payload.
•	PATCH /api/me/dashboard/active-event — сохранить пользовательский выбор активного события.
8.6. Рекомендуемый состав payload workspace
•	event summary
•	my memberships by role
•	my team summary
•	pending invitations
•	missing profile fields
•	missing event form fields
•	deadline widgets
•	timeline / notices
•	available quick actions
•	calendar preview
•	support/contact info
8.7. UX и производительность
•	Главный экран кабинета должен быть usable за 1–2 секунды и не собираться из N+1 запросов.
•	Состояние dashboard должно быть оптимистично обновляемым после локальных действий.
•	На мобильных экранах селектор события и быстрые действия должны оставаться доступными без длинного скролла.
9. Команды и полномочия капитана
Командный контур уже существует и должен быть доработан эволюционно, а не переписан. Главная правка: капитан может управлять составом и базовыми данными команды только до момента сабмита заявки/команды.
9.1. До сабмита команда находится в editable draft state
•	Капитан может изменить название команды.
•	Капитан может изменить описание команды.
•	Капитан может удалить участника команды.
•	Капитан может отклонить pending member request.
•	Капитан может отменить email invitation.
•	Капитан может передать капитанство.
•	Капитан может видеть заполненность слотов и блокеры перед сабмитом.
9.2. После сабмита команда фиксируется
После действия submitTeamForApproval команда должна перейти в фиксированное состояние. Бизнес-правило меняется по сравнению с более мягкой текущей моделью: блокировка наступает уже после сабмита, а не только после окончательного approve.
•	Нельзя менять название и описание.
•	Нельзя удалять участников.
•	Нельзя добавлять новых участников стандартным способом.
•	Нельзя менять капитана стандартным способом, кроме специально разрешённого административного сценария.
•	Нельзя менять приглашения и слоты.
9.3. Разрешённые сценарии после фиксации
•	Администратор может вернуть команду в статус CHANGES_REQUIRED / DRAFT_REOPENED с указанием причины.
•	Если проект уже использует EventTeamChangeRequest, его нужно задействовать как controlled post-submit edit flow.
•	Капитан в locked state может только просматривать состав, видеть причину блокировки и при необходимости создать запрос на изменение, если такой режим разрешён настройкой события.
9.4. Новые статусы и правила
При необходимости расширить статусы EventTeamStatus, не ломая текущую схему. Допустимы варианты DRAFT, SUBMITTED, CHANGES_REQUIRED, APPROVED, REJECTED, ARCHIVED. Если сохраняется текущий набор статусов, нужно явно описать серверные guard conditions для PENDING/ACTIVE/CHANGES_PENDING.
9.5. API и валидация
•	PATCH /api/events/:id/teams/:teamId — разрешён только до сабмита.
•	DELETE /api/events/:id/teams/:teamId/members/:userId — разрешён только до сабмита.
•	POST /api/events/:id/teams/:teamId/submit — переводит команду в locked submitted state.
•	При любой запрещённой операции возвращать 409 TEAM_SUBMITTED_LOCKED или аналогичный код.
9.6. UI
•	На карточке команды явно показывать: Draft / Submitted / Needs changes / Approved / Rejected.
•	После сабмита вместо активных кнопок показывать причину блокировки и доступные следующие шаги.
